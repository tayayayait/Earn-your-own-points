# Admin Integrations

Phase 4-2 adds the administrator API integration route:

- `src/routes/_authenticated/admin.integrations.tsx`
- `src/features/admin-integrations/integration-data.ts`
- `supabase/migrations/20260703160535_phase4_admin_integrations.sql`

## Scope

The page includes four sections:

- API Key management
- Webhook settings
- Integration status
- Failure logs

API keys are generated by Supabase RPCs. The full key is returned only once on create or regenerate. The UI describes this as one-time display. Stored rows keep only `key_hash`, `key_prefix`, and `key_suffix`.

Webhook signing keys are returned only once on create or rotation. The page later shows only prefix/suffix masking.

## Supabase RPCs

`public.get_admin_integrations()` returns:

- `api_keys`
- `webhooks`
- `status`
- `failure_logs`

Mutation RPCs:

- `public.create_admin_api_key(_name, _reason)`
- `public.regenerate_admin_api_key(_api_key_id, _reason)`
- `public.revoke_admin_api_key(_api_key_id, _reason)`
- `public.save_admin_webhook(_webhook_id, _name, _url, _event_types, _status, _rotate_signing_key, _reason)`
- `public.test_admin_webhook(_webhook_id, _reason)`
- `public.retry_admin_webhook_log(_log_id, _reason)`

All RPCs use `SECURITY DEFINER`, check `public.has_role((select auth.uid()), 'admin')`, revoke public execution, and grant execution only to `authenticated`.

## Webhook Test Send

This phase records a Supabase-managed test log through `test_admin_webhook`.

Actual outbound HTTP delivery is not implemented in this phase because the implementation plan lists external integration scope as an open question. A future Edge Function can perform the real HTTP POST and then reuse `webhook_logs` for status and retry tracking.
