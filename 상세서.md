# 포인트 적립 솔루션 UI/UX 상세서

문서 유형: Reference / Implementation Spec  
문서 상태: 구현 전 기준 문서  
적용 범위: Java Spring Boot 백엔드, Vue.js SPA 프론트엔드, 관리자 화면, 사용자 화면

## 1. 목적

이 문서는 포인트 적립 솔루션을 실제 구현할 때 개발자와 디자이너가 같은 기준으로 화면, 흐름, 상태, 컴포넌트, 예외 처리를 만들기 위한 기준 문서다. 추상 표현보다 수치, 규칙, 조건, 화면 예시를 우선한다.

## 2. 전제 및 미정 항목

| 항목 | 기준 |
|---|---|
| 기본 언어 | 한국어 |
| 프론트엔드 | Vue.js SPA |
| 백엔드 | Java / Spring Boot REST API |
| 인증 방식 | JWT 또는 세션 인증 중 최종 방식 확실한 정보 없음. UI는 토큰 만료와 재로그인 흐름을 지원해야 한다. |
| 브랜드 로고 | 확실한 정보 없음. 기본 텍스트 로고를 사용하고, 고객사 로고 업로드 후 교체한다. |
| 실제 포인트 명칭 | 확실한 정보 없음. 기본 명칭은 `포인트`이며 설정에서 `마일리지`, `스탬프`, `리워드` 등으로 변경 가능해야 한다. |
| 결제/POS 연동사 | 확실한 정보 없음. 외부 연동 화면은 API Key, Webhook URL, 연동 상태, 실패 로그를 표시하는 구조로 설계한다. |

## 3. 사용자 역할

| 역할 | 권한 범위 | 진입 화면 |
|---|---|---|
| 고객 | 본인 포인트 조회, 내역 조회, 사용 가능 조건 확인, 프로필 수정 | 사용자 홈 |
| 매장 운영자 | 고객 조회, 포인트 수동 적립/차감, 거래 내역 조회 | 관리자 대시보드 |
| 정책 관리자 | 포인트 정책, 등급 정책, 이벤트 정책 생성/수정 | 정책 관리 |
| 최고 관리자 | 관리자 계정, 권한, 브랜드 설정, API 연동 설정, 감사 로그 조회 | 관리자 설정 |
| 읽기 전용 관리자 | 모든 통계와 내역 조회, 수정 불가 | 관리자 대시보드 |

## 4. 정보 구조

### 4-1. 관리자 메뉴

관리자 화면은 좌측 사이드바, 상단 헤더, 본문 영역으로 구성한다.

| 1차 메뉴 | 2차 메뉴 | 기본 경로 | 주요 기능 |
|---|---|---|---|
| 대시보드 | 없음 | `/admin/dashboard` | KPI, 최근 거래, 랭킹, 이벤트 현황 |
| 고객 관리 | 고객 목록 | `/admin/customers` | 검색, 필터, 고객 상태 변경 |
| 고객 관리 | 고객 상세 | `/admin/customers/:customerId` | 기본정보, 포인트 잔액, 거래 이력 |
| 포인트 거래 | 거래 내역 | `/admin/transactions` | 적립, 사용, 취소, 만료, 조정 내역 |
| 포인트 거래 | 수동 지급/차감 | `/admin/transactions/manual` | 관리자 수동 처리 |
| 정책 관리 | 기본 정책 | `/admin/policies/base` | 기본 적립률, 사용 조건 |
| 정책 관리 | 등급 정책 | `/admin/policies/tiers` | 등급별 적립률, 승급 조건 |
| 정책 관리 | 상품/카테고리 정책 | `/admin/policies/products` | 상품별, 카테고리별 예외 정책 |
| 이벤트 관리 | 이벤트 목록 | `/admin/events` | 이벤트 포인트 설정 |
| 통계/리포트 | 기간별 통계 | `/admin/reports` | 차트, CSV 다운로드 |
| 연동 관리 | API 연동 | `/admin/integrations` | API Key, Webhook, 실패 로그 |
| 설정 | 브랜드 설정 | `/admin/settings/brand` | 로고, 컬러, 포인트 명칭 |
| 설정 | 관리자/권한 | `/admin/settings/admins` | 계정, 역할, 메뉴 권한 |
| 설정 | 감사 로그 | `/admin/settings/audit-logs` | 관리자 변경 이력 |

### 4-2. 사용자 메뉴

사용자 화면은 모바일 우선 구조로 만든다. 데스크톱에서는 상단 탭을 사용하고, 모바일에서는 하단 내비게이션을 사용한다.

| 메뉴 | 기본 경로 | 주요 기능 |
|---|---|---|
| 홈 | `/app/home` | 보유 포인트, 사용 가능 포인트, 예정 포인트 |
| 내역 | `/app/transactions` | 적립/사용/만료 내역 조회 |
| 혜택 | `/app/benefits` | 이벤트, 등급 혜택 |
| 내 정보 | `/app/profile` | 기본정보, 마케팅 수신, 탈퇴 요청 |

## 5. 공통 레이아웃 규칙

### 5-1. 화면 크기 기준

| 구간 | 너비 | 처리 기준 |
|---|---:|---|
| Mobile | 0-639px | 사용자 화면 기본 기준. 관리자 테이블은 카드 리스트로 전환한다. |
| Tablet | 640-1023px | 관리자 사이드바는 접힘 상태, 필터는 2열 배치한다. |
| Desktop | 1024-1439px | 관리자 기본 화면. 사이드바 고정, 본문 최대 너비 제한 없음. |
| Wide | 1440px 이상 | 대시보드는 12컬럼 그리드, 표는 전체 폭 사용. |

### 5-2. 관리자 레이아웃

| 영역 | 규격 |
|---|---|
| 사이드바 | 기본 240px, 접힘 72px, 배경 `#0F172A`, 텍스트 `#CBD5E1` |
| 상단 헤더 | 높이 56px, 흰색 배경, 하단 보더 `#E2E8F0` |
| 본문 | 최소 높이 `calc(100vh - 56px)`, 배경 `#F8FAFC`, 패딩 24px |
| 콘텐츠 폭 | 1024px 이상에서는 가로폭 전체 사용, 1440px 이상에서는 차트 행만 12컬럼 |
| 페이지 헤더 | 제목, 설명, 주요 액션 버튼, 보조 액션 버튼 순서 |
| 테이블 영역 | 필터 바, 액션 바, 테이블, 페이지네이션 순서 |

### 5-3. 사용자 레이아웃

| 영역 | Mobile | Desktop |
|---|---|---|
| 상단 영역 | 높이 56px, 뒤로가기/제목/알림 | 높이 64px, 로고/탭/프로필 |
| 본문 패딩 | 16px | 24px |
| 주요 카드 | 한 줄 1개 | 최대 3열 |
| 하단 내비게이션 | 표시, 높이 64px | 숨김 |
| 본문 최대 폭 | 제한 없음 | 960px 중앙 정렬 |

### 5-4. 간격 체계

기본 간격 단위는 4px이다.

| 토큰 | 값 | 용도 |
|---|---:|---|
| `space-1` | 4px | 아이콘과 텍스트 사이 |
| `space-2` | 8px | 작은 요소 내부 간격 |
| `space-3` | 12px | 폼 필드 간격, 테이블 셀 좌우 |
| `space-4` | 16px | 카드 내부 기본 패딩 |
| `space-6` | 24px | 섹션 간격, 관리자 본문 패딩 |
| `space-8` | 32px | 페이지 헤더와 본문 간격 |
| `space-10` | 40px | 대형 섹션 간격 |

### 6-3. 브랜드 커스터마이징 규칙

1. 고객사 브랜드 색상은 `primary-600`, `primary-700`, `primary-50`만 교체한다.
2. 적립, 차감, 오류, 경고 색상은 브랜드 색상으로 교체하지 않는다.
3. 로고는 SVG 또는 PNG를 허용한다. 권장 크기는 관리자 헤더 160x40px, 사용자 헤더 120x32px이다.
4. 텍스트 대비는 WCAG AA 기준을 따른다. 일반 텍스트는 4.5:1 이상, 18px 이상 굵은 텍스트는 3:1 이상이어야 한다.

## 7. 폰트 체계

### 7-1. 기본 폰트

```css
font-family: Pretendard, "Noto Sans KR", "Apple SD Gothic Neo", Arial, sans-serif;
```

숫자 금액, 포인트, 통계 값은 `font-variant-numeric: tabular-nums;`를 적용한다.

### 7-2. 타입 스케일

| 토큰 | 크기 | 줄 높이 | 굵기 | 용도 |
|---|---:|---:|---:|---|
| `display-sm` | 32px | 40px | 700 | 사용자 보유 포인트 숫자 |
| `heading-lg` | 24px | 32px | 700 | 페이지 제목 |
| `heading-md` | 20px | 28px | 700 | 모달 제목, 섹션 제목 |
| `heading-sm` | 18px | 26px | 600 | 카드 제목 |
| `body-md` | 14px | 22px | 400 | 기본 본문, 폼 입력 |
| `body-sm` | 13px | 20px | 400 | 테이블 보조 정보 |
| `caption` | 12px | 18px | 400 | 라벨, 도움말, 배지 |

규칙:

1. 버튼, 배지, 탭의 글자 간격은 `0`으로 고정한다.
2. 관리자 테이블 본문은 13px 또는 14px만 사용한다.
3. 금액과 포인트 숫자는 오른쪽 정렬한다.
4. 긴 한글 단어가 버튼을 넘치면 버튼 폭을 늘리고, 320px 미만 화면에서는 두 줄 허용 대신 아이콘 버튼으로 전환한다.

## 8. 공통 컴포넌트 규격

### 8-1. 버튼

| 크기 | 높이 | 좌우 패딩 | 폰트 | 사용 위치 |
|---|---:|---:|---|---|
| `sm` | 32px | 12px | 13px / 600 | 테이블 행 액션 |
| `md` | 40px | 16px | 14px / 600 | 일반 폼, 필터 |
| `lg` | 48px | 20px | 15px / 700 | 주요 저장, 사용자 CTA |

| Variant | 배경 | 텍스트 | 보더 | 사용 조건 |
|---|---|---|---|---|
| `primary` | `primary-600` | `white` | 없음 | 저장, 등록, 정책 활성화 |
| `secondary` | `white` | `slate-700` | `slate-200` | 취소, 보조 이동 |
| `ghost` | 투명 | `slate-700` | 없음 | 테이블 내 가벼운 액션 |
| `danger` | `danger-600` | `white` | 없음 | 차단, 삭제성 비활성화, 강제 취소 |
| `success` | `accent-500` | `white` | 없음 | 포인트 지급 확정 |

상태:

| 상태 | 처리 |
|---|---|
| 기본 | hover, active, focus 상태를 모두 제공한다. |
| Hover | primary는 `primary-700`, secondary는 `slate-50` 배경을 사용한다. |
| Focus | 2px ring `primary-600`, 바깥 offset 2px를 적용한다. |
| Disabled | opacity 0.45, 클릭 불가, tooltip에 비활성 사유 표시 가능 |
| Loading | 왼쪽에 16px spinner, 라벨은 유지한다. 버튼 폭은 변하지 않아야 한다. |

### 8-2. 아이콘 버튼

| 속성 | 값 |
|---|---|
| 기본 크기 | 36x36px |
| 모바일 터치 영역 | 최소 44x44px |
| 아이콘 크기 | 18px |
| 라운드 | 6px |
| Tooltip | hover 및 focus 시 300ms 후 표시 |

아이콘 버튼은 편집, 삭제, 새로고침, 다운로드, 닫기, 뒤로가기, 필터 열기처럼 의미가 명확한 액션에 사용한다. 아이콘만 있는 버튼에는 `aria-label`을 필수로 지정한다.

### 8-3. 입력창

| 속성 | 값 |
|---|---|
| 높이 | 40px |
| 좌우 패딩 | 12px |
| 배경 | `white` |
| 보더 | 1px `slate-200` |
| 라운드 | 6px |
| 폰트 | 14px |
| Placeholder | `slate-400` |
| Focus | 보더 `primary-600`, ring `primary-50` 3px |
| Error | 보더 `danger-600`, 하단 메시지 `danger-600` 12px |

라벨은 입력창 위에 배치한다. 라벨과 입력창 간격은 6px, 필드와 필드 간격은 16px이다.

### 8-4. 셀렉트와 콤보박스

| 항목 수 | 컴포넌트 |
|---:|---|
| 1-7개 | Select |
| 8개 이상 | Searchable Combobox |
| 원격 검색 필요 | Async Combobox |

관리자 고객 검색, 상품 검색, 카테고리 검색은 Async Combobox를 사용한다. 입력 후 300ms debounce를 적용한다.

### 8-5. 카드

| 항목 | 규격 |
|---|---|
| 배경 | `white` |
| 보더 | 1px `slate-200` |
| 라운드 | 8px |
| 그림자 | 기본 사용 금지. hover 카드만 `0 4px 12px rgba(15, 23, 42, 0.06)` 허용 |
| 패딩 | 관리자 16px, 사용자 주요 카드 20px |
| 제목 | 14-18px, 카드 용도에 따라 제한 |

카드 안에 또 다른 카드 형태의 컨테이너를 넣지 않는다. 필요한 경우 구분선 또는 2컬럼 레이아웃을 사용한다.

### 8-6. 테이블

| 속성 | 값 |
|---|---|
| 헤더 높이 | 40px |
| 행 높이 | 44px 기본, 최대 56px |
| 셀 좌우 패딩 | 12px |
| 헤더 배경 | `slate-50` |
| 헤더 텍스트 | 12px / 600 / `slate-500` |
| 본문 텍스트 | 13px 또는 14px |
| 행 보더 | 1px `slate-100` |
| Hover | `primary-50` 40% 투명도 또는 `#F8FAFC` |
| Sticky Header | 20행 이상 목록에서 필수 |

정렬:

1. 고객명, 상태, 유형, 메모는 왼쪽 정렬한다.
2. 포인트, 금액, 비율, 건수는 오른쪽 정렬한다.
3. 날짜와 거래 ID는 고정폭 폰트 또는 tabular nums를 적용한다.
4. 행 클릭으로 상세 이동이 가능하면 hover 시 cursor pointer를 적용한다.

### 8-7. 필터 바

필터 바는 테이블 상단에 배치한다.

| 요소 | 규격 |
|---|---|
| 검색창 | 최소 280px, 최대 420px |
| 기간 선택 | 최근 7일, 30일, 이번 달, 직접 선택 |
| 상태 필터 | 다중 선택 가능 |
| 초기화 버튼 | 필터 적용 시에만 활성 |
| 검색 실행 | Enter 또는 검색 버튼 |

모바일에서는 필터 버튼 1개만 노출하고, 필터 항목은 하단 시트로 연다.

### 8-8. 모달

| 크기 | 폭 | 사용 예 |
|---|---:|---|
| `sm` | 400px | 확인, 경고 |
| `md` | 560px | 포인트 수동 지급, 상태 변경 |
| `lg` | 720px | 정책 상세 편집 |
| `xl` | 960px | 복합 조건 설정 |

규칙:

1. 모달 상단에는 제목과 닫기 버튼을 둔다.
2. 본문 최대 높이는 `calc(100vh - 180px)`로 제한하고 내부 스크롤을 허용한다.
3. 하단 액션은 오른쪽 정렬한다. 기본 순서는 `취소`, `확인`이다.
4. 위험 액션은 확인 문구 입력을 요구한다. 예: `차단` 입력 후 차단 버튼 활성화.
5. 640px 미만에서는 전체 화면 모달로 전환한다.

### 8-9. 토스트

| 유형 | 표시 시간 | 예시 |
|---|---:|---|
| 성공 | 3000ms | `포인트 지급이 완료되었습니다.` |
| 오류 | 사용자 닫기 전까지 또는 6000ms | `처리에 실패했습니다. 다시 시도하세요.` |
| 정보 | 4000ms | `정책 변경은 다음 주문부터 적용됩니다.` |

토스트는 화면 오른쪽 상단에 표시한다. 모바일에서는 하단 16px 위에 표시한다.

### 8-10. 배지

| 상태 | 라벨 | 색상 |
|---|---|---|
| 회원 활성 | `활성` | 적립 완료 색상 |
| 휴면 | `휴면` | 경고 색상 |
| 탈퇴 | `탈퇴` | 취소 색상 |
| 차단 | `차단` | 오류 색상 |
| 적립 예정 | `예정` | 경고 색상 |
| 확정 | `확정` | 적립 완료 색상 |
| 사용 | `사용` | 사용 완료 색상 |
| 실패 | `실패` | 오류 색상 |

배지는 24px 높이, 좌우 8px 패딩, 12px 폰트를 사용한다.

## 9. 화면별 상세 구조

### 9-1. 관리자 로그인

경로: `/admin/login`

목적: 관리자 인증 후 권한에 맞는 화면으로 이동한다.

구조:

| 순서 | 영역 | 구성 |
|---:|---|---|
| 1 | 브랜드 | 로고 또는 서비스명 |
| 2 | 로그인 폼 | 이메일, 비밀번호 |
| 3 | 보안 옵션 | 로그인 유지 체크박스 |
| 4 | 액션 | 로그인 버튼 |
| 5 | 보조 링크 | 비밀번호 재설정 |

검증:

| 필드 | 규칙 | 오류 문구 |
|---|---|---|
| 이메일 | 필수, 이메일 형식 | `이메일 형식이 올바르지 않습니다.` |

| 상황 | 처리 |

| 권한 없음 | 로그인 성공 후 `/admin/no-permission` 이동 |
| 5회 실패 | 10분 잠금, 남은 시간 표시 |

### 9-2. 관리자 대시보드

경로: `/admin/dashboard`

목적: 전체 포인트 운영 현황을 10초 이내에 파악한다.

상단 KPI 카드:

| 카드 | 값 | 보조 정보 | 계산 기준 |
|---|---|---|---|
| 누적 적립 포인트 | `123,456,000P` | 전월 대비 `+12.4%` | 확정 적립 합계 |
| 누적 사용 포인트 | `45,120,000P` | 전월 대비 `-3.2%` | 사용 확정 합계 |
| 잔여 포인트 총액 | `78,336,000P` | 활성 고객 기준 | 현재 사용 가능 잔액 합계 |
| 만료 예정 포인트 | `3,210,000P` | 30일 이내 | 만료일 기준 |

본문 레이아웃:

| 영역 | Desktop | Mobile/Tablet |
|---|---|---|
| KPI 카드 | 4열 | 1열 또는 2열 |
| 기간별 추이 차트 | 8컬럼 | 전체 폭 |
| 거래 유형 도넛 차트 | 4컬럼 | 전체 폭 |
| 최근 거래 | 8컬럼 | 카드 리스트 |
| 고객 랭킹 | 4컬럼 | 카드 리스트 |

차트 규칙:

1. 기본 기간은 최근 30일이다.
2. 기간 옵션은 `7일`, `30일`, `90일`, `이번 달`, `직접 선택`이다.
3. 적립은 녹색, 사용은 빨간색, 만료는 회색, 예정은 주황색으로 표시한다.
4. 차트 값은 hover tooltip에 날짜, 유형, 포인트, 건수를 표시한다.

빈 상태:

`선택한 기간에 포인트 거래가 없습니다.` 문구와 기간 초기화 버튼을 표시한다.

### 9-3. 고객 목록

경로: `/admin/customers`

목적: 고객을 빠르게 검색하고 상태, 보유 포인트, 최근 거래를 확인한다.

필터:

| 필터 | 타입 | 기본값 | 조건 |
|---|---|---|---|
| 검색어 | 텍스트 | 없음 | 이름, 휴대폰, 이메일, 고객 ID 검색 |
| 회원 상태 | 다중 선택 | 전체 | 활성, 휴면, 탈퇴, 차단 |
| 등급 | 다중 선택 | 전체 | 등급 정책에 등록된 값 |
| 포인트 범위 | 숫자 범위 | 없음 | 최소, 최대 |
| 가입일 | 날짜 범위 | 없음 | 직접 선택 가능 |

테이블 컬럼:

| 컬럼 | 폭 | 정렬 | 형식 |
|---|---:|---|---|
| 고객 ID | 120px | 좌측 | `CUS-000001` |
| 이름 | 140px | 좌측 | 이름 + 연락처 보조 텍스트 |
| 등급 | 96px | 좌측 | 배지 |
| 상태 | 80px | 좌측 | 배지 |
| 보유 포인트 | 140px | 우측 | `12,300P` |
| 적립 예정 | 120px | 우측 | `1,000P` |
| 최근 거래 | 160px | 좌측 | `2026-07-03 14:00` 형식 |
| 액션 | 120px | 우측 | 상세, 상태변경 |

행 액션:

1. `상세`: 고객 상세로 이동한다.
2. `지급`: 수동 지급 모달을 연다. 탈퇴/차단 고객은 비활성화한다.
3. `차단`: 활성/휴면 고객만 가능하다.

### 9-4. 고객 상세

경로: `/admin/customers/:customerId`

목적: 고객 단위 포인트 상태와 변경 이력을 확인하고 필요한 조치를 수행한다.

상단 구조:

| 영역 | 표시 정보 |
|---|---|
| 고객 요약 | 이름, 고객 ID, 등급, 상태, 가입일 |
| 주요 액션 | 정보 수정, 포인트 지급, 포인트 차감, 상태 변경 |
| 포인트 요약 | 사용 가능, 적립 예정, 만료 예정, 누적 적립, 누적 사용 |

탭:

| 탭 | 내용 |
|---|---|
| 기본정보 | 이름, 연락처, 이메일, 생년월일, 마케팅 수신 |
| 포인트 내역 | 적립, 사용, 취소, 만료, 관리자 조정 |
| 주문/외부 거래 | 외부 주문 ID, POS 거래 ID |
| 관리자 메모 | 내부 메모, 변경 이력 |

수정 규칙:

1. 고객 상태가 `탈퇴`이면 개인정보 수정 버튼을 숨긴다.
2. 고객 상태가 `차단`이면 포인트 지급/사용 액션을 비활성화한다.
3. 관리자 메모는 삭제하지 않고 새 메모를 누적한다.
4. 포인트 조정은 사유 입력이 필수이며 10자 이상이어야 한다.

### 9-5. 포인트 거래 내역

경로: `/admin/transactions`

목적: 모든 포인트 변경을 거래 단위로 추적한다.

필터:

| 필터 | 타입 | 기준 |
|---|---|---|
| 거래 ID | 텍스트 | 정확히 일치 |
| 고객 | Async Combobox | 이름, 휴대폰, 고객 ID |
| 거래 유형 | 다중 선택 | 적립, 사용, 취소, 만료, 조정, 이벤트 |
| 상태 | 다중 선택 | 예정, 확정, 취소, 실패 |
| 기간 | 날짜 범위 | 생성일 기준 |
| 외부 거래 ID | 텍스트 | 주문/POS 거래 추적 |

테이블 컬럼:

| 컬럼 | 정렬 | 형식 |
|---|---|---|
| 거래 ID | 좌측 | `PTX-20260703-000001` |
| 고객 | 좌측 | 이름 + 고객 ID |
| 유형 | 좌측 | 배지 |
| 상태 | 좌측 | 배지 |
| 포인트 | 우측 | 적립 `+1,000P`, 사용 `-500P` |
| 잔액 | 우측 | 처리 후 잔액 |
| 사유 | 좌측 | 주문 적립, 이벤트 지급, 관리자 조정 |
| 외부 거래 ID | 좌측 | 주문/POS ID |
| 생성일 | 좌측 | `YYYY-MM-DD HH:mm` |
| 액션 | 우측 | 상세, 취소, 재처리 |

거래 상세 패널:

1. 기본 거래 정보
2. 고객 정보
3. 정책 적용 정보
4. 원본 요청 payload 요약
5. 처리 로그
6. 취소/재처리 가능 여부

### 9-6. 수동 포인트 지급/차감

경로: `/admin/transactions/manual` 또는 고객 상세 모달

폼 필드:

| 필드 | 타입 | 필수 | 규칙 |
|---|---|---|---|
| 고객 | Async Combobox | 예 | 활성 고객만 선택 가능 |
| 처리 유형 | Segmented Control | 예 | 지급, 차감 |
| 포인트 | 숫자 입력 | 예 | 1 이상, 정수만 허용 |
| 적용일 | 날짜/시간 | 예 | 기본 현재 시각 |
| 만료일 | 날짜 | 지급 시 조건부 | 기본 정책 만료일 자동 계산 |
| 사유 | textarea | 예 | 10-500자 |
| 내부 메모 | textarea | 아니오 | 500자 이하 |

검증:

| 조건 | 처리 |
|---|---|
| 차감 포인트가 사용 가능 포인트보다 큼 | 저장 비활성, `보유 포인트보다 많이 차감할 수 없습니다.` |
| 차단/탈퇴 고객 | 고객 선택 불가 |
| 사유 10자 미만 | inline error |
| 중복 제출 | idempotency key 생성, 버튼 loading 유지 |

확인 모달:

`고객명`, `처리 유형`, `포인트`, `처리 후 예상 잔액`, `사유`를 표시하고 최종 확인을 요구한다.

### 9-7. 기본 정책 관리

경로: `/admin/policies/base`

목적: 서비스 전체 기본 적립률, 사용 조건, 만료 조건을 설정한다.

섹션:

| 섹션 | 필드 | 규칙 |
|---|---|---|
| 적립 기준 | 기본 적립률 | 0-100%, 소수점 2자리까지 |
| 적립 기준 | 적립 단위 | 1P, 10P, 100P 중 선택 |
| 적립 기준 | 반올림 방식 | 버림, 반올림, 올림 |
| 사용 조건 | 최소 사용 포인트 | 0 이상 정수 |
| 사용 조건 | 최대 사용 비율 | 주문금액의 0-100% |
| 사용 조건 | 사용 단위 | 1P, 10P, 100P |
| 만료 조건 | 유효기간 | 적립 확정일 기준 N개월 |
| 확정 조건 | 적립 예정 기간 | 구매 후 N일 뒤 확정 |
| 제외 조건 | 적립 제외 결제수단 | 쿠폰, 포인트 결제분 등 |

저장 규칙:

1. 저장 전 변경 diff를 표시한다.
2. 즉시 적용과 예약 적용을 선택할 수 있다.
3. 예약 적용 시 시작일시는 현재 시각 이후여야 한다.
4. 활성 정책은 삭제할 수 없다. 비활성화만 가능하다.
5. 정책 변경 이력은 감사 로그에 자동 기록한다.

### 9-8. 등급 정책 관리

경로: `/admin/policies/tiers`

목적: 회원 등급별 적립률과 혜택을 설정한다.

테이블 컬럼:

| 컬럼 | 예시 |
|---|---|
| 등급명 | Silver, Gold, VIP |
| 승급 기준 | 최근 6개월 구매금액 500,000원 이상 |
| 기본 적립률 | 3.0% |
| 추가 적립률 | +1.0% |
| 최소 유지 조건 | 최근 6개월 구매 1회 이상 |
| 상태 | 활성, 비활성 |

편집 규칙:

1. 등급 순서는 드래그로 변경 가능하다.
2. 같은 등급명이 중복되면 저장할 수 없다.
3. 승급 기준 기간은 1-24개월 사이로 제한한다.
4. 등급 삭제 시 해당 고객 처리 방식을 선택해야 한다. 옵션은 `하위 등급 이동`, `기본 등급 이동`이다.

### 9-9. 상품/카테고리 정책

경로: `/admin/policies/products`

목적: 특정 상품 또는 카테고리별 적립률을 기본 정책보다 우선 적용한다.

우선순위:

1. 상품 단위 정책
2. 카테고리 단위 정책
3. 이벤트 정책
4. 등급 정책
5. 기본 정책

동일 우선순위 정책이 겹치면 최신 생성 정책이 아니라 `우선순위 값`이 높은 정책을 적용한다. 우선순위 값이 같으면 저장을 차단한다.

필드:

| 필드 | 타입 | 규칙 |
|---|---|---|
| 정책명 | 텍스트 | 2-50자 |
| 대상 유형 | Radio | 상품, 카테고리 |
| 대상 | Async Combobox | 다중 선택 가능 |
| 적립률 | 숫자 | 0-100% |
| 적용 기간 | 날짜 범위 | 종료일 없음 허용 |
| 우선순위 | 숫자 | 1-999 |
| 제외 여부 | Toggle | 켜면 적립률 입력 비활성 |

### 9-10. 이벤트 포인트 관리

경로: `/admin/events`

목적: 기간별, 고객군별, 상품별 이벤트 포인트를 관리한다.

이벤트 목록 컬럼:

| 컬럼 | 형식 |
|---|---|
| 이벤트명 | 텍스트 |
| 상태 | 예정, 진행, 일시중지, 종료 |
| 기간 | `YYYY-MM-DD HH:mm - YYYY-MM-DD HH:mm` |
| 대상 | 전체, 등급, 고객군, 상품 |
| 지급 방식 | 추가 적립률, 고정 포인트 |
| 지급 한도 | 고객당, 전체 한도 |
| 지급 현황 | 지급 건수 / 예산 |

이벤트 생성 단계:

1. 기본정보: 이벤트명, 설명, 기간
2. 대상 설정: 고객 등급, 고객 세그먼트, 상품/카테고리
3. 지급 방식: 추가 적립률 또는 고정 포인트
4. 한도 설정: 고객당 최대 포인트, 전체 예산
5. 검토: 예상 지급 조건과 충돌 정책 표시

충돌 처리:

| 충돌 | 처리 |
|---|---|
| 이벤트 기간 중복 | 경고 표시, 우선순위 설정 요구 |
| 전체 예산 없음 | 저장 가능, 단 확인 모달에 경고 |
| 고객당 한도 0 | 저장 불가 |
| 종료일이 시작일보다 빠름 | 저장 불가 |

### 9-12. API 연동 관리

경로: `/admin/integrations`

목적: 외부 쇼핑몰, POS, 결제 시스템과 포인트 API를 연동한다.

구조:

| 섹션 | 구성 |
|---|---|
| API Key | 키 이름, 생성일, 마지막 사용일, 상태, 재발급 |
| Webhook | URL, 이벤트 유형, 서명 검증 키, 테스트 전송 |
| 연동 상태 | 성공률, 최근 실패, 평균 응답시간 |
| 실패 로그 | 요청 ID, 엔드포인트, 오류 코드, 재시도 |

API Key 규칙:

1. 생성 직후 1회만 전체 키를 표시한다.
2. 목록에는 앞 6자와 뒤 4자만 표시한다. 예: `pk_123456****9abc`
3. 재발급 시 기존 키는 즉시 비활성 또는 예약 비활성을 선택한다.
4. 삭제 대신 비활성화 상태를 사용한다.

Webhook 테스트:

1. 테스트 payload를 표시한다.
2. 응답 코드, 응답 시간, 응답 body 첫 2KB를 표시한다.
3. 실패 시 TLS 오류, DNS 오류, 4xx, 5xx를 구분한다.

## 11. 핵심 사용자 흐름

### 11-1. 구매 적립 흐름

1. 외부 주문/POS 시스템이 적립 API를 호출한다.
2. 백엔드는 고객 상태를 확인한다.
3. 정책 우선순위에 따라 적립 포인트를 계산한다.
4. 확정 전 기간이 있으면 거래 상태를 `PENDING`으로 생성한다.
5. 고객 홈에는 `적립 예정`으로 표시한다.
6. 확정일이 지나면 상태를 `CONFIRMED`로 변경한다.
7. 고객 사용 가능 포인트에 반영한다.

UI 표시:

| 상태 | 고객 화면 | 관리자 화면 |
|---|---|---|
| PENDING | `적립 예정` | 거래 상태 `예정`, 취소 가능 |
| CONFIRMED | `적립 완료` | 거래 상태 `확정`, 사용 가능 |
| CANCELED | `적립 취소` | 원 거래와 취소 거래 모두 표시 |
| FAILED | 고객 화면 미표시 | 실패 로그와 재처리 버튼 표시 |

### 11-2. 포인트 사용 흐름

1. 고객 또는 외부 결제 시스템이 사용 가능 포인트를 조회한다.
2. 사용 요청 시 최소 사용 포인트와 최대 사용 비율을 검증한다.
3. 사용할 포인트를 예약 또는 즉시 차감한다.
4. 결제 성공 시 `USE_CONFIRMED` 거래를 생성한다.
5. 결제 실패 또는 취소 시 포인트를 복구한다.

검증 순서:

1. 고객 상태가 활성인지 확인
2. 사용 가능 포인트가 요청 포인트 이상인지 확인
3. 최소 사용 포인트 이상인지 확인
4. 주문금액 대비 최대 사용 비율 이하인지 확인
5. 만료일이 빠른 포인트부터 차감

### 11-3. 포인트 취소 흐름

1. 원 거래 ID를 입력하거나 외부 거래 ID로 원 거래를 찾는다.
2. 원 거래 상태가 취소 가능한지 확인한다.
3. 적립 취소는 사용된 포인트 여부를 확인한다.
4. 사용 취소는 차감된 포인트를 복구한다.
5. 취소 거래를 별도 거래 ID로 생성한다.

취소 가능 조건:

| 원 거래 유형 | 취소 가능 상태 | 제한 |
|---|---|---|
| 적립 예정 | PENDING | 즉시 취소 가능 |
| 적립 확정 | CONFIRMED | 이미 사용된 포인트가 있으면 부분 취소 또는 실패 |
| 사용 확정 | CONFIRMED | 주문 취소 확인 후 복구 |
| 만료 | EXPIRED | 관리자 권한으로만 복구 |
| 관리자 조정 | CONFIRMED | OWNER 또는 MANAGER만 취소 |

### 11-4. 정책 변경 흐름

1. 정책 관리자가 정책 값을 수정한다.
2. 저장 전 변경 diff를 확인한다.
3. 즉시 적용 또는 예약 적용을 선택한다.
4. 적용 시각 이후 신규 거래부터 새 정책을 적용한다.
5. 기존 거래 재계산은 자동 수행하지 않는다.

기존 거래 재계산이 필요한 경우 별도 배치 작업으로 처리하고, UI에서 예상 영향 건수를 먼저 표시한다.

### 11-5. 이벤트 지급 흐름

1. 이벤트를 생성하고 대상과 기간을 설정한다.
2. 이벤트 상태가 `SCHEDULED`가 된다.
3. 시작 시간이 되면 `RUNNING`으로 변경한다.
4. 조건에 맞는 거래 발생 시 이벤트 포인트를 추가 지급한다.
5. 예산 또는 고객 한도에 도달하면 해당 대상 지급을 중지한다.
6. 종료 시간이 되면 `ENDED`로 변경한다.

| `CANCELED` | 취소 | 원 거래 취소 완료 |
| `FAILED` | 실패 | 처리 실패, 재처리 가능 여부 표시 |
| `EXPIRED` | 만료 | 유효기간 만료 처리 완료 |

### 12-4. 정책 상태

| 값 | 라벨 | 설명 |
|---|---|---|
| `DRAFT` | 초안 | 저장 전 또는 임시 저장 |
| `SCHEDULED` | 예약 | 미래 시점 적용 예정 |
| `ACTIVE` | 활성 | 현재 적용 중 |
| `PAUSED` | 일시중지 | 일시적으로 적용 중단 |
| `ENDED` | 종료 | 적용 기간 종료 |
| `DISABLED` | 비활성 | 관리자가 비활성화 |

### 12-5. 비동기 작업 상태

| 값 | 라벨 | UI 처리 |
|---|---|---|
| `QUEUED` | 대기 | 작업 목록에 대기 표시 |
| `RUNNING` | 처리 중 | 진행률 또는 spinner |
| `SUCCEEDED` | 완료 | 결과 다운로드 또는 상세 링크 |
| `FAILED` | 실패 | 오류 사유와 재시도 버튼 |
| `RETRYING` | 재시도 중 | 재시도 횟수 표시 |

## 13. 데이터 표시 규칙

### 13-1. 포인트 형식

| 상황 | 형식 |
|---|---|
| 기본 | `12,300P` |
| 적립 | `+12,300P` |
| 사용/만료/차감 | `-12,300P` |
| 0 포인트 | `0P` |
| 값 없음 | `-` |

포인트 명칭이 변경되면 접미사 `P` 대신 설정한 명칭을 사용한다. 예: `12,300마일`.

### 13-2. 날짜 형식

| 위치 | 형식 |
|---|---|
| 관리자 테이블 | `YYYY-MM-DD HH:mm` |
| 사용자 화면 | `YYYY.MM.DD HH:mm` |
| 짧은 날짜 | `YYYY.MM.DD` |
| 기간 | `YYYY.MM.DD - YYYY.MM.DD` |

시간대는 서버 기준 시간대를 사용한다. 시간대 정책이 미정이면 사용자에게 시간대 선택 UI를 제공하지 않는다.

### 13-3. 전화번호와 이메일

| 데이터 | 표시 |
|---|---|
| 휴대폰 | `010-1234-5678` |
| 관리자 목록 보조 텍스트 | `010-****-5678` |
| 이메일 | 원문 표시 |
| 개인정보 마스킹 화면 | `ab***@domain.com` |

## 14. API 연동 UI 기준

프론트엔드는 API 응답을 다음 형태로 해석할 수 있어야 한다.

```json
{
  "success": false,
  "code": "POINT_INSUFFICIENT_BALANCE",
  "message": "사용 가능한 포인트가 부족합니다.",
  "traceId": "req_20260703_000001",
  "details": {
    "availablePoint": 1200,
    "requestedPoint": 2000
  }
}
```

오류 응답 표시 규칙:

| 오류 코드 | 사용자 표시 | 관리자 표시 |
|---|---|---|
| `AUTH_REQUIRED` | `로그인이 필요합니다.` | 동일 |
| `PERMISSION_DENIED` | `권한이 없습니다.` | 필요한 권한 표시 |
| `POINT_INSUFFICIENT_BALANCE` | `사용 가능한 포인트가 부족합니다.` | 요청/보유 포인트 표시 |
| `POLICY_CONFLICT` | `적용할 수 없는 정책 조건입니다.` | 충돌 정책 ID 표시 |
| `DUPLICATE_TRANSACTION` | `이미 처리된 거래입니다.` | 기존 거래 ID 링크 |
| `EXTERNAL_API_FAILED` | `외부 연동 처리에 실패했습니다.` | traceId, 재시도 버튼 |
| `VALIDATION_ERROR` | 필드 하단 오류 | 필드별 오류 목록 |

traceId는 관리자 화면 오류 상세와 토스트 세부 정보에 표시한다. 고객 화면에는 traceId를 기본 숨김 처리하고 고객센터 문의 시 복사할 수 있게 한다.

## 15. 접근성 원칙

| 항목 | 기준 |
|---|---|
| 키보드 접근 | 모든 버튼, 링크, 입력, 탭, 모달은 Tab으로 접근 가능해야 한다. |
| Focus 표시 | 2px 이상 명확한 focus ring을 제공한다. |
| 대비 | 일반 텍스트 4.5:1 이상, 큰 텍스트 3:1 이상 |

| 스크린리더 | 아이콘 버튼, 차트, 상태 배지에 aria-label을 제공한다. |
| 모달 | 열릴 때 첫 입력 또는 제목에 focus, 닫힐 때 트리거 버튼으로 focus 복귀 |
| 오류 메시지 | 입력 필드와 `aria-describedby`로 연결한다. |
| 터치 영역 | 모바일 클릭 대상은 최소 44x44px |
| 애니메이션 | 필수 정보 전달에 애니메이션을 사용하지 않는다. |

차트 접근성:

1. 차트 아래에 동일 데이터를 요약한 테이블 또는 다운로드 옵션을 제공한다.
2. 색상만으로 적립/사용을 구분하지 않는다.
3. tooltip 정보는 키보드 focus로도 접근 가능해야 한다.

## 18. 보안 및 감사 UI 기준

1. 포인트 거래는 UI에서 삭제 기능을 제공하지 않는다.
2. 취소와 정정은 새 거래를 생성해 원 거래와 연결한다.
3. 관리자 수동 지급/차감에는 사유가 필수다.
4. 위험 액션은 확인 모달을 사용한다.
5. API Key 전체 값은 생성 직후 1회만 표시한다.
6. 감사 로그는 읽기 전용이다.
7. 개인정보 표시 화면은 권한에 따라 마스킹한다.
8. CSV 다운로드는 관리자 ID와 필터 조건을 감사 로그에 기록한다.
